Frontend Starter Contract

The default Carbide UI uses React, Bun, and Tailwind in a container while Carbide stays focused on the monorepo, runtime, and API contract.

Runtime Model

browser -> web container -> /api proxy -> api Go container -> db

The web service is the public entrypoint. It proxies /api and /health to the API service so browser requests stay same-origin.

Bun frontend, Go API backend, Postgres database remains the generated app baseline.

The web app lives inside web. It is TypeScript-first: main.tsx owns browser route state and API calls, then renders the starter screens.

component/l1, component/l2, and component/l3 mirror the Tailwind organization model: primitives, composed patterns, and product screens.

DashboardLayout is a conventional app shell with a left sidebar, section navigation, account footer, and main work area.

Carbide's component-design taste should track Tailwind Plus and Catalyst: default to Application UI patterns for product work, keep components production-ready, fully responsive, accessible, and easy to customize, keep styling in utility classes directly in markup, favor neutral operational layouts over decorative chrome, include normal interaction states, and prefer conventional primitives such as sidebars, stacked headers, form sections, cards, tables, dialogs, dropdowns, and stats blocks.

Styling

The starter uses Tailwind as the required styling path. The web container installs with Bun and builds generated CSS from the checked-in Tailwind input.

styles.css contains only the Tailwind import, TypeScript-aware source directives, and the data-theme dark variant. It does not own global html/body sizing, layout defaults, scrollbar pseudo-selectors, or a generated color-variable palette. tokens.ts contains reusable Tailwind utility groups, including starter light/dark visual choices and built-in scrollbar styling. The scaffold does not add a parallel theme.css file.

carbide health rejects global html/body sizing, custom selectors, scrollbar pseudo-selectors, --carbide-* color variables, @theme, @apply, @layer, keyframes, media rules, and container rules in styles.css; component styling belongs in Tailwind utility classes and component class layers.

Theme mode follows one starter pattern: ThemeToggle.tsx lives in component/l1, the selected mode stays browser-local, localStorage stores that mode, system resolves through matchMedia('(prefers-color-scheme: dark)'), the app sets document.documentElement.dataset.theme to the resolved light or dark value, sets document.documentElement.dataset.themeMode to the selected mode, mirrors the resolved mode through document.documentElement.style.colorScheme, and threads mode, resolved, and onMode through L2/L3 components instead of recreating theme state in multiple places.

Use tokens.ts for reusable class groups, compose variants in TypeScript, and keep responsive layout, typography, spacing, colors, and scrollbars in Tailwind utilities. If a real product intentionally needs global CSS, create web/src/product.css, import it explicitly, document the reason in local app docs if they exist, and update the law contract.

typecheck runs tsc --noEmit. Docker builds run typecheck before building browser assets.

Browser Assets

Generated apps cache-bust React browser assets by default. assets:build runs Tailwind, runs Bun's browser build with content-hashed filenames under web/public/assets/, and writes web/public/index.html plus web/public/asset-manifest.json.

The HTML app shell and asset manifest are served with Cache-Control: no-store. Files under /assets/ are served with Cache-Control: public, max-age=31536000, immutable.

web/public/ and web/src/tailwind.css are generated build output and are ignored in generated app repositories.

Regression Requirements

  • Generated apps include web, API, and db containers.
  • The Bun web service proxies /api and /health to the API service.
  • Auth uses same-origin cookies without CORS setup.
  • The React app shell references content-hashed JS and CSS assets.
  • The app shell and asset manifest are served no-store.
  • Hashed assets are served with one-year immutable caching.
  • Frontend installs with bun install --frozen-lockfile and builds with bun run build.
  • Tailwind is required in the generated web app.
  • The generated web app includes component/l1, component/l2, and component/l3 starter tiers.
  • The tiers stay small: primitives in L1, reusable patterns in L2, product screens in L3.
