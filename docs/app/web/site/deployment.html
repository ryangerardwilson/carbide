Deployment

Carbide is not opinionated about production topology. It owns the local Docker dev loop and lets each app own production deploy behavior.

Deploy Model

Deploy targets are checked in under deploy.targets in carbide.toml. Each target points to an app-owned script. Carbide resolves the target name and runs that script from the app root.

carbide deploy prod

Generated apps ship with no deploy target by default. That keeps production ownership with the app instead of the framework.

Target Contract

[deploy.targets.prod]
script = "./deploy/prod.sh"
description = "Deploy the production environment."

Target names use lowercase letters, numbers, and dashes. Script paths stay inside the app repo. The script may call SSH, rsync, Docker Compose, Terraform, Kubernetes tools, platform CLIs, or anything else the app needs.

Script Ownership

Carbide does not interpret SSH topology, nginx, Cloudflare, multi-VM orchestration, or any other production platform detail anymore. The app-owned script handles those choices.

This keeps the framework contract small:

  • Carbide owns local Docker dev fidelity.
  • The app owns production deploy behavior.
  • carbide.toml gives the app one declarative entrypoint per deploy target.

Script Environment

  • CARBIDE_DEPLOY_TARGET: target name passed by the CLI.
  • CARBIDE_PROJECT_ROOT: absolute app root path.
  • CARBIDE_PROJECT_NAME: app display name when available.
  • CARBIDE_PROJECT_SLUG: app slug when available.
