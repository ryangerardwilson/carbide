Create Your First App

The first Carbide app should boot with one command and include a Bun/React/Tailwind web service, Go API backend, Postgres-backed account creation, login, logout, and dashboard flow by default.

Install, create, run, register is the intended first-session path.

Happy Path

curl -fsSL https://raw.githubusercontent.com/ryangerardwilson/carbide/main/cli/install.sh | bash
carbide new demo
cd demo
carbide run dev
carbide status
carbide health
carbide stop dev

The installer uses a release binary when available and falls back to a source build. Go is required only for that source-build fallback. The generated app does not require host Bun, Node, Go API setup, or Postgres because those run inside containers.

Open the app URL printed by carbide run dev. The command prefers port 8080 and silently selects another local port when 8080 is already in use. To choose one explicitly, run CARBIDE_HTTP_PORT=18080 carbide run dev.

The web service listens on port 8080 inside its container. Your browser should use the host URL printed by the CLI.

Run Dev

carbide run dev starts the stack with quiet Compose output, watch enabled when supported, and live logs streamed below the startup summary. Edits under web/src/, web package/config files, api/, db/, or api/Dockerfile rebuild and replace the relevant container.

The CLI presents aligned rows and compact tables with TTY-only color and full-width terminal-only per-container startup and shutdown animation while Docker Compose builds, starts, waits for, or stops containers. Captured or piped output remains plain text for tests, scripts, and AI agents.

Before startup, carbide run dev prints only the working app and API URLs. Logs begin only after Compose reports the stack ready. Ctrl+C detaches from live log streaming and leaves the containers running.

Web, API, db, and watch events appear in one timestamped, service-tagged stream and are mirrored to .carbide/log/dev.jsonl. NO_COLOR disables ANSI color without disabling the terminal startup or shutdown animation.

If local session state is unclear after a detach or failed startup, run carbide clean dev before restarting the stack. It stops project containers and orphans without deleting volumes.

Generated App

  • Bun/React/Tailwind web container.
  • Go API container.
  • Postgres service container.
  • Checked-in Docker Compose infrastructure.
  • Register, login, logout, and dashboard routes.
  • api/ and db/ directories for API and database code.
  • Postgres-backed users and sessions.
  • Queryable structured dev logs.

The generated app starts with no seeded users. The first browser visit opens account creation. Registration creates the first user and session; later sessions use the login form.

Commands

  • carbide help prints the command reference.
  • carbide upgrade upgrades the installed CLI.
  • carbide audit clears any previous .audit/, renders the current starter into .audit/starter-reference/, creates one markdown report per law and taste slice under .audit/report/, and in an interactive terminal with codex installed spawns those report audits through the Codex CLI.
  • carbide resolve reads .audit/report/*.md, writes .audit/plan.md, and asks for clarification only when the reports surface a real ambiguity.
  • carbide fix implements the ready .audit/plan.md through the Codex CLI and writes .audit/fix.md.
  • carbide new <project-name> creates a new project directory. Human names are accepted: carbide new "My Carbide App" creates my-carbide-app, stores name = "My Carbide App", and stores slug = "my-carbide-app".
  • carbide init initializes the current empty directory.
  • carbide run dev starts the Docker Compose dev stack, prints the app URL and API URL, then streams logs until Ctrl+C detaches.
  • carbide clean dev normalizes local dev state before a fresh restart without deleting volumes.
  • carbide status prints services, containers, published ports, internal ports, and status.
  • carbide health checks app laws: project shape, required config, checked-in deploy-script targets, Compose, env/secrets, file line limits, and legacy-regression markers.
  • carbide health runtime starts the Docker stack if needed, verifies health, registration, dashboard, and logout, then stops containers it started.
  • carbide stop dev stops the Docker Compose dev stack.
  • carbide logs reads .carbide/log/dev.jsonl with service, text, limit, and JSON queries. carbide follow logs attaches to live container logs.
  • Machine-readable state uses command-shaped JSON subcommands such as carbide status json, carbide health json, and carbide health runtime json, not --json flags.

Troubleshooting

  • If carbide health fails, fix the named contract before changing runtime behavior. Use carbide health json or carbide health env json when an agent needs structured output.
  • If container state is unclear after Ctrl+C, run carbide clean dev, then carbide run dev.
  • If deploy fails, inspect the app-owned script registered under [deploy.targets.*] in carbide.toml. Carbide only resolves the target name and runs that script.
  • If the user wants the one-shot path, run carbide audit resolve fix. Run carbide audit, carbide resolve, and carbide fix separately when the user wants to review the intermediate report and plan. carbide health framework is for the Carbide source repo itself, not for generated apps.
